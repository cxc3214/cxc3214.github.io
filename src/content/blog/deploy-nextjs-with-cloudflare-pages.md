Cloudflare Pages 是我很喜欢的静态站点和前端项目部署平台。它的优势很明显：全球 CDN、自动构建、预览部署、自定义域名、HTTPS 证书和相对简单的控制台体验。对于 Next.js 项目来说，如果目标是静态导出或以边缘方式运行一部分能力，Cloudflare Pages 是一个值得考虑的选择。不过部署之前，需要先确认项目使用的 Next.js 特性是否适合当前平台。

第一步是判断项目形态。如果你的 Next.js 项目主要是静态页面、Markdown 内容、产品介绍页、文档站或不依赖传统 Node.js 服务端能力的页面，那么迁移到 Cloudflare Pages 通常比较顺畅。如果项目大量使用 Node.js API、文件系统写入、长连接、复杂服务端中间件或依赖特定 Node 运行时，就需要更仔细地评估。部署平台不是越强越好，而是要和项目运行方式匹配。

对于静态导出项目，常见做法是在 `next.config.js` 中配置 `output: "export"`，让 Next.js 在构建时生成静态文件。构建命令通常是 `npm run build`，输出目录一般是 `out`。在 Cloudflare Pages 里连接 GitHub 仓库后，选择对应分支，设置构建命令和输出目录即可。这个流程最简单，适合博客、落地页、作品集和不需要服务端渲染的工具页面。

如果项目使用了 App Router，也要注意哪些页面可以静态生成，哪些页面需要动态能力。一个常见误区是本地开发时所有页面都能跑，就以为部署平台也一定支持。实际上，本地开发服务器包含了很多开发期能力，而静态部署要求页面在构建时就能确定输出。凡是依赖请求时用户状态、服务端实时查询或动态路径生成的部分，都要明确它的运行模型。

第二步是处理环境变量。Cloudflare Pages 支持在项目设置中配置生产环境和预览环境的变量。需要区分公开变量和服务端变量。以 Next.js 为例，前端可见的变量通常带有 `NEXT_PUBLIC_` 前缀，它们会被打包到浏览器代码中，不适合放密钥。API Token、数据库密码、私有服务地址等敏感信息不要暴露给前端。如果项目只是静态导出，很多所谓的“服务端变量”其实在构建时就会被读取，这也意味着构建日志和产物安全需要一起考虑。

第三步是关注依赖安装和 Node 版本。Cloudflare Pages 会根据仓库里的 lock 文件安装依赖，所以建议提交 `package-lock.json`、`pnpm-lock.yaml` 或 `yarn.lock`。Node 版本可以通过控制台环境变量或项目配置指定，避免本地和线上版本差异导致构建失败。尤其是 Next.js、Astro、Vite 等工具链升级较快，版本不一致时很容易出现本地正常、线上失败的情况。

第四步是绑定自定义域名。在 Pages 项目中添加自定义域名后，Cloudflare 会提示需要配置 DNS。如果域名托管在 Cloudflare，通常可以自动添加记录；如果不在 Cloudflare，就要去域名服务商处手动添加 CNAME 或其他记录。绑定成功后，Cloudflare 会自动签发 HTTPS 证书。证书生效可能需要几分钟到几十分钟，刚配置完不要急着反复改记录。

部署 Next.js 时，我会特别关注几个常见问题。第一，构建输出目录是否正确。如果平台期待 `out`，但项目实际输出到 `.next`，部署结果可能是空页面。第二，路由刷新是否正常。静态站点访问深层路径时，需要确认平台能正确回退到对应文件。第三，图片优化是否依赖 Next.js 服务端。如果使用了 `next/image` 的默认优化能力，在纯静态导出下可能需要配置 `unoptimized: true` 或换成普通图片处理方式。第四，接口请求是否跨域。前端部署到新域名后，调用旧 API 经常会遇到 CORS 或 Cookie 域名问题。

我倾向于把部署流程写进 README，而不是只记在脑子里。一个项目只要超过一个人维护，或者未来自己隔几个月还会回来改，就应该写清楚：本地启动命令是什么，构建命令是什么，输出目录是什么，环境变量有哪些，域名如何配置，出现构建失败时先看哪里。这些文档看起来不产生功能，却能显著降低维护成本。

Cloudflare Pages 的另一个好处是预览部署。每次 Pull Request 或分支提交都可以生成一个临时访问地址，这对于检查页面、分享给别人确认、测试 SEO 元信息都很方便。正式发布前，我通常会看首页、关键页面、移动端布局、控制台错误、网络请求和构建日志。不要只看页面“能打开”，还要看是否存在 404、资源加载失败、环境变量缺失和错误的 canonical。

总结一下，用 Cloudflare Pages 部署 Next.js 项目的核心不是记住某个按钮，而是弄清楚项目的运行模型。如果是静态导出，就让构建产物清晰、输出目录正确、域名和缓存配置稳定。如果需要动态能力，就提前确认平台支持方式。部署本身只是最后一步，真正影响稳定性的，是你在部署前对项目边界、依赖、环境变量和域名的整理。
